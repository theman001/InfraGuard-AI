# 한국 법 자문 서비스 - 전체 프로젝트 정리

> 최초 작성: 2026.04.09  
> 상태: 구상 단계

---

## 📌 프로젝트 개요

**목적:** 기업 보안 담당자(1인)가 정책팀 역할을 수행할 수 있도록 법령 기반 자문을 자동화하는 웹 서비스

**핵심 가치:**
- 법령 검토 시간 수 시간 → 수 분으로 단축
- AI가 아닌 **법령과 판례에 근거한 문서화** 가능
- 혼자서 정책팀 역할 실질적 대체
- 과거 자문 이력 기반 맥락 연속성 확보

---

## 🏗️ 시스템 아키텍처

```
[사용자 - 웹 브라우저]
        ↓ 포트포워딩 (로컬/외부 접근)
[Streamlit 웹 앱 - 로컬 실행]
        │
        ├── 계정 관리 (로그인/회원가입)
        ├── 템플릿 관리
        ├── 환경 설정값 입력
        ├── 자문 요청 입력
        ├── history 참고 옵션 ON/OFF
        │
        ├── ChromaDB (법령 RAG)
        ├── 대법원 API (판례 실시간)
        ├── Claude API (자문 생성)
        └── DB (계정 / 보고서 / 히스토리)
                ↓
        자문 보고서 출력 + PDF 저장
```

## 📂 핵심 기능 구성

### 1. 법령 RAG

- **수집:** 국가법령정보센터 Open API
- **저장:** ChromaDB (로컬 벡터 DB)
- **업데이트:** 매월 1일 자동 실행 (APScheduler)
- **변경 감지 방식:**
  - 최초 수집 시 법령별 시행일자/개정일자를 DB에 저장
  - 매월 1일 API 재조회 → 저장된 날짜와 비교
  - 날짜가 다른 법령만 텍스트 재수집 + ChromaDB 업데이트
  - 날짜가 동일한 법령은 스킵

- **청킹 방식:** 조(Article) 단위
  - 조 전체를 하나의 청크로 저장
  - 메타데이터 구성:
    ```
    {
      법령명: "개인정보보호법",
      조번호: "제15조",
      조제목: "개인정보의 수집·이용",
      업종태그: ["공통"] or ["금융", "의료" ...],
      시행일자: "2023-09-15",
      개정일자: "2023-09-15"
    }
    ```

| 업종 | 추가 법령 |
|---|---|

| 의료/헬스케어 | 의료법, 의료기기법, 생명윤리 및 안전에 관한 법률 |
| 제조 | 산업안전보건법, 중대재해 처벌 등에 관한 법률 |
| 공공/공급망 | 국가정보보호법, 소프트웨어 진흥법 |
| 유통/이커머스 | 전자상거래 등에서의 소비자보호에 관한 법률, 표시·광고의 공정화에 관한 법률 |

**인증별 추가 (템플릿에서 인증 선택 시 자동 포함)**

| 인증 | 추가 법령/기준 |
|---|---|
| ISMS-P | 개인정보보호법 시행령·시행규칙, 정보통신망법 시행령 |
| ISO 27001 | 국제표준 연계 국내법 매핑 (개인정보보호법, 정보통신망법) |

### 2. 판례 연동

- **방식:** RAG 구축 X → 실시간 API 호출
- **출처:** 대법원 판례 Open API
- **시점:** 자문 요청 시마다 실시간 검색
- **키워드 생성 방식:** LLM이 사전 생성
  - 자문 요청 프롬프트를 LLM이 먼저 읽고 판례 검색에 적합한 키워드 생성
  - 생성된 키워드로 대법원 API 호출
  - 예) 사용자 입력: "재택근무자 개인 디바이스 허용 정책"
        → LLM 생성 키워드: "BYOD 개인정보 수집 동의"
  - API 호출 흐름:
    ```
    1. 사용자 요청 입력
    2. LLM → 판례 검색 키워드 생성 (API 호출 1회)
    3. 대법원 API → 판례 검색 (키워드 기반)
    4. 판례 결과 + RAG 법령 결과 → LLM 최종 자문 생성
    ```

### 3. 법령 우선순위 원칙 및 회색지대 처리

#### 기본 3원칙

| 원칙 | 내용 | 적용 강도 |
|---|---|---|
| 상위법 > 하위법 | 헌법 > 법률 > 시행령 > 시행규칙 | 절대적 (예외 없음) |
| 특별법 > 일반법 | 특정 대상/상황을 규율하는 법이 우선 | 조건부 |
| 신법 > 구법 | 나중에 제정·개정된 법이 우선 | 조건부 |

#### LLM이 판단 가능한 케이스 (자문 진행)

```
Case 1. 상위법 vs 하위법
─────────────────────────────────────
시행령이 법률 조항에 위배되는 경우
→ 법률 조항 우선 적용, 시행령 무효 명시

Case 2. 특별법이 명확히 해당 사안을 규정
─────────────────────────────────────
예) 금융회사의 개인정보 처리
→ 개인정보보호법(일반법) < 신용정보법(특별법) 우선 적용

Case 3. 동일 사안에 대해 신법이 명시적으로 개정
─────────────────────────────────────
예) 데이터 3법 개정(2020) 이후 정보통신망법 개인정보 조항
→ 개인정보보호법으로 이관된 조항은 개인정보보호법 적용
```

#### LLM이 판단하면 안 되는 케이스 (회색지대 출력)

```
Case 4. 특별법(구법) vs 일반법(신법) 충돌
─────────────────────────────────────
특별법 우선 원칙 vs 신법 우선 원칙이 동시에 충돌
→ ⚠️ "법령 충돌 / 해석 분분 / 외부 자문 권고" 출력

Case 5. 부칙·경과규정이 있는 경우
─────────────────────────────────────
법령 개정 시 부칙에서 구법 적용 기간을 별도 규정
→ ⚠️ "경과규정 존재 / 적용 시점 확인 필요 / 외부 자문 권고" 출력

Case 6. 판례가 원칙을 뒤집은 경우
─────────────────────────────────────
법리상 원칙과 다른 방향으로 판례가 형성된 경우
→ ⚠️ "판례 상충 / 해석 분분 / 외부 자문 권고" 출력

Case 7. 관련 판례 자체가 없는 경우
─────────────────────────────────────
신규 법령·기술·서비스로 판례가 미형성된 영역
→ ⚠️ "판례 없음 / 외부 자문 권고" 출력
```

```
Case 8. Case 1~7에 해당하지 않는 모든 경우 (폴백)
─────────────────────────────────────
관련 법령 자체를 RAG에서 찾지 못한 경우,
또는 어떤 케이스에도 분류되지 않는 예외 상황
→ ⚠️ "적용 법령 불명확 / 해당 법령 부재 또는 확인 불가 / 외부 자문 권고" 출력
→ 자문 결과 생성 중단, 보고서에 사유 명시
```

#### 회색지대 판단 기준 (System Prompt에 명시)

```
아래 조건 중 하나라도 해당되면 회색지대로 분류:
1. 두 개 이상의 법령 우선순위 원칙이 동시에 충돌
2. 법령에 부칙·경과규정이 존재
3. 검색된 판례가 서로 상반된 결론
4. 관련 판례가 존재하지 않는 신규 영역
5. 법제처 공식 해석례가 없는 영역
6. Case 1~7 어디에도 분류되지 않는 경우 (폴백)
   → 반드시 외부 자문 권고 출력, 임의 판단 금지
```

### 4. 프롬프트 구성

```
System Prompt (고정 - 결과값 제어)
─────────────────────────────────────
- 출력 형식 지정
- 반드시 법령 조항 명시
- 요청 한줄요약 생성 지시
- 유사 history 선별 지시 (없으면 빈값)
- [history 참고 옵션 ON] 시 과거 맥락 반영 지시
- 면책 고지 포함
- 법령 우선순위 3원칙 적용 지시
- 회색지대 처리 로직 (Case 4~7 해당 시 외부 자문 권고 출력)

User Prompt (요청마다 조합)
─────────────────────────────────────
템플릿 값 (회사 컨텍스트)
+ 환경 설정값 (현재 상황 키워드)
+ RAG 검색 결과 (관련 법령 조항 자동 삽입)
+ 자문 요청 프롬프트 (사용자 입력)
+ history 리스트 (한줄요약 + URL 목록)
+ history 참고 옵션 ON/OFF 값
```

### 5. 템플릿 시스템

- **목적:** 요청마다 반복 입력하는 고정 컨텍스트 저장
- **사용 방식:** 저장된 템플릿 불러오기 → 키워드 + 프롬프트 추가 입력
- **포함 항목 및 선택값:**

| 항목 | 선택값 | 비고 |
|---|---|---|
| 회사 규모 | 대기업 / 중견 / 중소 / 스타트업 | |
| 업종 | IT·소프트웨어 / 금융·핀테크 / 제조 / 의료·헬스케어 / 공공·공급망 / 유통·이커머스 / 기타 | 업종별 추가 법령 자동 매핑 |
| 보유 인증 | ISMS-P / ISO 27001 / 둘 다 / 없음 | 인증별 추가 법령 자동 매핑 |
| 개인정보 처리 여부 | 처리함 / 처리 안 함 | |
| 임직원 수 | 50명 이하 / 51~300명 / 300명 초과 | 법령 적용 기준 차등 |
| 개인정보 처리 규모 | 1만명 미만 / 1만~100만명 / 100만명 초과 | 개인정보보호법 의무 수준 차등 |
| 해외 서비스 여부 | 있음 / 없음 | GDPR 등 해외 법령 적용 여부 |
| 상장 여부 | 상장 / 비상장 | 공시 의무 관련 법령 적용 여부 |
| 주요 데이터 유형 | 개인정보 / 민감정보 / 기술자료 / 복합 | 적용 법령 범위 결정 |

- **항목별 "사용 안함" 옵션:**
  - 각 항목마다 "사용 안함" 체크박스 제공
  - 체크 시 해당 항목은 프롬프트에 포함시키지 않고 전송
  - 예) 해외 서비스 여부 "사용 안함" 체크 → User Prompt에서 해당 항목 제외

- **템플릿 개수:** 계정당 제한 없음

- **수정·삭제 정책:**
  - 수정: 자유롭게 가능
  - 삭제: 삭제 시 팝업으로 아래 두 가지 선택지 제공
    ```
    이 템플릿을 삭제하시겠습니까?
    ○ 템플릿만 삭제 (연결된 보고서 이력은 유지)
    ○ 템플릿 + 연결된 보고서 이력 모두 삭제
    ```

### 5. 히스토리 시스템

```
자문 요청 시 흐름
─────────────────────────────────────
1. 해당 계정의 history 리스트 (한줄요약 + URL) 불러오기
2. Claude API에 함께 전달
3. Claude가 판단
   ├── 자문 결과 생성
   ├── 요청 한줄요약 생성
   └── 유사 history 선별 (URL 포함)
4. 결과 저장
   ├── 보고서 저장 (URL 생성)
   └── history에 한줄요약 + URL 추가
```

**history 참고 옵션 동작**

| 옵션 | Claude 동작 |
|---|---|
| ON | 유사 history 내용을 맥락으로 반영해서 자문 |
| OFF | 유사 history는 비고로만 제공, 판단에는 미반영 |
| 공통 | 유사 history 리스트는 항상 비고에 포함 |

**유사 히스토리 선별 기준**
- **LLM 전달 범위:** 계정의 최근 100건 한줄요약 + URL 목록만 전달
- **유사도 판단:** LLM이 한줄요약 의미 기반으로 유사 항목 선별
- **비고 표시 개수:** 유사 히스토리 최대 5개까지 표시
- **100건 초과 시:** 오래된 항목부터 제외하고 최근 100건만 유지

**히스토리 보관 및 삭제 정책**
- **보관 기간:** 제한 없음 (자동 삭제 없음)
- **사용자 직접 삭제:** 가능
  - 삭제 시 연결된 보고서도 함께 삭제
  - 삭제 전 확인 팝업 제공
    ```
    이 히스토리를 삭제하시겠습니까?
    삭제 시 연결된 보고서도 함께 삭제됩니다.
    [확인] [취소]
    ```

### 6. 자문 보고서 출력

```
─────────────────────────────────────
```

## 🗄️ DB 구조

```
users
├── user_id
├── email
├── password (hashed)
├── role              ← "admin" / "user"
├── status            ← "pending" / "active" / "suspended"
└── created_at

templates
├── template_id
├── user_id
├── template_name
├── created_at
├── updated_at
└── items (JSON)     ← 각 항목값 + 사용 안함 여부 포함

reports
├── report_id        ← URL용
├── user_id
├── template_id      ← 연결된 템플릿 ID (삭제 시 NULL 처리)
├── template_snapshot (JSON) ← 보고서 생성 당시 템플릿 전문 스냅샷
├── created_at
├── summary          ← 한줄요약
├── prompt           ← 사용자 입력 프롬프트
└── result (full)    ← 보고서 전문
```

**계정 권한 구조**

| 권한 | 기능 |
|---|---|
| 관리자 (admin) | 회원가입 승인/거절, 계정 정지/활성화, 전체 계정 목록 조회 |
| 일반 사용자 (user) | 자문 요청, 템플릿 관리, 본인 히스토리/보고서 조회·삭제 |

**회원가입 플로우**
```
사용자 회원가입 요청
        ↓
status = "pending" 으로 저장
        ↓
관리자 승인 → status = "active" (로그인 가능)
관리자 거절 → status = "suspended" (로그인 불가)
```

## 🗓️ 구현 단계

```
1단계 - 법령 수집 + ChromaDB 구축
├── 국가법령정보센터 API 키 발급 (✅ 완료)
├── 대상 법령 리스트 정의 (✅ 완료 - 기본 공통 10개 + 업종별/인증별 추가)
├── 법령 텍스트 수집 스크립트
├── 조(Article) 단위 청킹 + 메타데이터 구성
├── 임베딩 + ChromaDB 저장
└── 매월 1일 자동 업데이트 스케줄러 (시행일자/개정일자 기반 변경 감지)

2단계 - Claude API 연동 + 자문 기능
├── System Prompt 설계 (결과값 제어)
├── User Prompt 조합 로직
│   (템플릿 + 환경값 + RAG결과 + 요청 + history)
├── 한줄요약 생성 로직
├── 유사 history 선별 로직
├── history 참고 옵션 ON/OFF 처리
├── 대법원 API 판례 실시간 호출 연동
└── 회색지대 처리 로직
   (대법원 API 키 발급 - 2단계 전 준비)

3단계 - Streamlit UI 구성
├── 계정 관리 (로그인/회원가입)
├── 템플릿 생성 / 불러오기 화면
├── 자문 요청 입력 화면
│   └── history 참고 옵션 토글
├── 보고서 출력 화면
└── 히스토리 목록 화면

4단계 - 보고서 고도화
├── PDF 다운로드
├── 보고서 이력 저장
└── 면책 고지 자동 포함

5단계 - 확장 (선택)
├── ISMS-P 버전 템플릿
├── ISO 27001 버전 템플릿
└── 인증 심사 기준별 체크리스트 자동 생성
```
